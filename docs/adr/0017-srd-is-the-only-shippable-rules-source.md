# SRD 5.1/5.2.1 are the only shippable rules source — PHB is a lookup aid, homebrew is the escape hatch

This project runs a Curse of Strahd table from the group's own translated PHB. That copy sits in the maintainer's working directory — excluded from the repo, never committed — and it is genuinely the most convenient thing to read when checking a rule. That convenience is a trap: the SRD is CC-BY-4.0 open content we may redistribute (see `NOTICE.md`), and the PHB is not. This ADR fixes the boundary so it stops being re-decided per feature.

The rule is not new in spirit — CLAUDE.md's Bottom line already said "Ship SRD/open data only in the repo," ADR-0005 keeps `source: "custom"` enemies local-only, and ADR-0015 states "character data in this repo means SRD open content." What was missing was that the same boundary governs **rules lookups, design-review citations, and which features we agree to build at all**, not just which data files get committed.

## Decision

1. **Lookup order is always SRD first.** Every rules question starts at the SRD text — a local clone of the CC-BY SRD kept with the maintainer's tooling, outside this repo. (Its exact location and the grep pattern live in the local agent notes; agents that cannot find it must ask rather than substitute another source.)
2. **The PHB may be consulted, but never concluded from.** It is a lookup aid — for rules the SRD omits, and for the Traditional Chinese wording the UI uses. Anything found there must then be confirmed to exist in the SRD before it informs code, a spec, or a report. The PHB material is excluded from the repo; nothing from it is ever committed, quoted into shipped strings, or transcribed into a seed file.
3. **Citations name SRD.** Design reviews, ADRs, and code comments cite `SRD <path> § <heading>`. `PHB p.###` does not appear in project artifacts, even local-only ones — the citation is what a future reader will act on, and it must point at the source we can stand behind.
4. **PHB-only content is a homebrew request, not a backlog item.** If a rule, subclass, species, spell, or monster exists only outside the SRD, we do not build it in. The player or DM enters it by hand through the manual-entry paths that already exist (ADR-0002's editable everything, ADR-0005's `source: "custom"`).

## Why

- **Licensing is the binding constraint, and it is not negotiable by convenience.** The SRD's CC-BY grant is what makes the repo publishable at all (ADR-0014 puts a public URL in the README). A single PHB-derived table or verbatim rules paragraph would compromise that for the sake of one feature.
- **"Reference it now, source it later" does not survive contact with a codebase.** A number transcribed from the PHB into a data file loses its provenance immediately; six months later nobody can tell which rows are safe. Requiring SRD confirmation *at lookup time* is the only checkpoint that reliably exists.
- **The manual-override architecture already pays for this.** ADR-0002 exists precisely so the engine never has to know every rule. A DM who wants a non-SRD subclass is not blocked — they are in the same position as one running homebrew, which this project was built to support. The licensing boundary costs the user very little because the escape hatch is the product's core feature.
- **The failure mode is quiet, so it needs a standing rule rather than review vigilance.** On 2026-07-20 a design review cited PHB page numbers throughout — not because the SRD lacked the rules (it had every one of them), but because the reviewer looked for the SRD clone in the wrong place, did not find it, and fell through to the PHB extract sitting ready to hand. Nothing in the tooling objected. The whole report had to be re-sourced.

## Considered options

- **Fold this into ADR-0015 (playground / character portability).** Rejected — 0015 already carries the narrower "character data in this repo means SRD open content," but this rule governs the enemy DB, the builder's class/species/background tables, spell and weapon metadata, design-review citations, and feature triage. Filed under a playground-mode ADR, nobody looking for a content-licensing rule would find it. 0015's line stays as-is and now reads as an instance of this ADR.
- **Fold into ADR-0005 (enemy DB).** Rejected for the same reason, narrower still — 0005's `"custom"`-is-local-only rule is one application of the boundary, not the boundary.
- **Delete the PHB material from the working copy entirely.** Rejected — it is gitignored, it is the DM's own legally-purchased translated copy, and it is the only source for the Chinese wording the table actually speaks. Removing it would push rule lookups to memory, which is worse: recalled 5e rules drift on exactly the edge cases reviews are for.
- **Allow PHB citations in local-only working docs** (they never leave the machine, so nothing leaks). Rejected — those docs are the specs implementation runs from. A PHB citation there is a PHB rule entering the product one step later, with the provenance already stripped.

## Consequence for future work

The local agent notes carry the operational version of this rule — lookup order, where the SRD clone actually lives, the grep pattern — since that is the file agents scan before touching code; this ADR is the durable *why* behind it. Any review tooling that offers a "fall back to the PHB" step is **overridden for this project**: fall back to the PHB to *find* something, never to *cite* it.

**SRD 5.2.1 (2024) support is planned, not hypothetical.** The current content baseline is 2014 / SRD 5.1, and everything shipped today is sourced from it — but 5.2.1 is also CC-BY open content, its text is already available alongside 5.1 in the same local clone, and the intent is to support it. So: **an agent encountering 2024 material (weapon mastery, the 2024 exhaustion track, 2024 species/backgrounds, a `5.2` edition tag) should not treat it as a mistake or "clean it up" back to 2014.** Check when it arrived and what it is labelled, then leave it alone.

**The intended adoption shape is rolling and opt-in per content area, driven by 5.1's thinness — not a migration.** SRD 5.1 ships very little character-origin content; backgrounds are the clearest case, with exactly **one** (Acolyte). SRD 5.2.1 has four (`04_Character_Origins/Backgrounds/Backgrounds_Each/`: Acolyte, Criminal, Sage, Soldier). So the plan is to pull 5.2.1 content in **area by area, wherever the 2014 SRD is too thin to play with**, each area gated behind an explicit **edition category** in the UI: a dropdown where the player picks 5.2.1 before the 5.2.1 options are even visible. Nothing is auto-mixed and nothing is silently upgraded — choosing the edition is a deliberate act by the person building the character, which is the UI-level expression of the no-silent-mixing invariant below. **None of this is built yet** (as of 2026-07-20) and it is not a backlog commitment with a date; it is the recorded direction so that the first 5.2.1 category to appear reads as intentional.

The known hard part, for whoever does it: 2024 backgrounds are not drop-in content. They carry an ability-score increase and a starting feat that 2014 backgrounds don't, so a 5.2.1 background on an otherwise-2014 character is an edition mix at the *mechanics* level, not just the data level. Deciding what a 5.2.1 category actually grants on a 2014 sheet is the real design work — the dropdown is the easy half.

What stays invariant is that the two editions are **never mixed silently**. 2024 rules differ from 2014 in ways that produce wrong numbers at the table if half-applied, so every 5.2-sourced rule must be identifiable as such — cited `SRD 5.2.1 <path> § <heading>` in specs and ADRs, and, once both editions coexist in shipped data, tagged per item rather than inferred from context. Until that tagging exists, adopting a 2024 rule is a deliberate, stated change; after it exists, the rule is simply "say which edition, always." Both editions are equally shippable under §Decision above — this paragraph is about correctness, not licensing.

If a non-SRD feature is ever genuinely worth building, the reversal is not "cite the PHB in a spec" — it is obtaining a license, or reimplementing the mechanic from an open source. Treat any PR that adds non-SRD rules content as a licensing change, not a feature.
