# Enemy DB: global table, lossless v.any() action blocks, DM-gated, deep-copy spawn

The enemy database (issue #6) is a single global `enemies` Convex table — no `gameId`, mirroring `characters` — holding three sources under one shape: `source: "seed"` (the 29-creature original gothic bestiary), `"srd"` (334 SRD 2014 monsters), and `"custom"` (DM-entered homebrew/CoS, local DB only, never shipped). Seeded via `scripts/gen-enemies.mjs` → `convex/enemySeed.ts` → `enemies.seedAll`, idempotent by `seedKey` — the same pipeline shape as `gen-characters`/`characterSeed`, and like the character cards, the in-app DB is the source of truth after seeding (seeded entries are editable; re-running seedAll never resets them).

**The five action blocks (`traits`/`actions`/`bonusActions`/`reactions`/`legendaryActions`) are `v.array(v.any())`, deliberately.** The two seed sources use incompatible per-action JSON shapes (bestiary: `kind`/`to_hit`/`damage:"2d4+2 穿刺"`/`on_hit`/`save`/`dc`/`on_fail`/`on_success`/`trigger`/`cost`/`attacks`; SRD: `desc`/`attack_bonus`/`damage:[{damage_dice,damage_type}]`/`dc:{dc_type,dc_value}`), and the acceptance criterion was lossless schema preservation so actions can keep becoming recipes as the model grows. A structured validator would either lose fields or grow to ~18 optionals that drift from the sources. Do not "tighten" these to typed validators without checking both source shapes round-trip — the lossless-round-trip test in `convex-tests/enemies.test.ts` is the guard. The one place that interprets actions is `recipeFromAction` in `convex/enemies.ts`, which dual-reads both shapes and returns null for anything unresolvable (multiattack/special/summon stay dmNotes reference text — never block, ADR-0002).

**Everything is DM-gated on the backend** (`requireDm`: `list`/`create`/`update`/`remove`/`seedAll`/`spawn`) — enemy stat blocks are Backstage secrets, consistent with the `projectCombatants` privacy split. The UI honors this by wiring `EnemyDbPanel` only in `Backstage.tsx` (passed to `GameBoard` as the optional `enemyDbPanel` ReactNode, same pattern as `dmPanel`); Frontstage has no wiring at all, which also sidesteps the two-page payload-mapping trap.

**Spawn is a deep copy with no back-reference**: template → independent `combatants` row (full HP, auto color/order, resist/vuln/immune split from the text fields, traits/tactics folded into dmNotes) + combatant-owned `recipes` rows from resolvable actions (attack: `to_hit`/`attack_bonus` + parsed damage; save: DC + zh ability via `saveAbilityToZh`, critImmune). Since the on-field enemy editor (2026-07-06), spawn also snapshots the full template (minus `seedKey`) onto the combatant as `statBlock` — still no back-reference; the snapshot is the instance's own editable copy, DM-only in the projection like dmNotes, and edited via the same `EnemyForm` reused in the 👹 window (saving it re-derives name/AC/maxHp/R-V-I with spawn's derivation; editing 動作 JSON never regenerates recipes). The shared field set lives in `convex/enemyFields.ts`, a server-free module so the frontend can import the type + `splitTypes` without pulling `_generated/server`. Editing an instance never touches the template and vice versa (ADR-0002); tested in both directions.

**Canonical damage types & condition immunity (2026-07-06, `d5a0923`).** Damage-type strings are normalized through a zh↔en alias map (`穿刺`/`Piercing` → `piercing`) in `convex/rules.ts` (`canonicalDamageType`), stripping qualifiers like （非魔法）/"from nonmagical weapons" so R/V/I chips and incoming attacks match regardless of source spelling; unrecognized fragments keep their raw text (inert but visible). `applyDamageType` compares both sides through the canonical form; `CombatantSheet` renders zh labels via `damageTypeLabel`. A new `conditionImmune` field is derived one-way from the stat-block 狀態免疫 text by `parseConditionImmunities` (`convex/enemyFields.ts`) at spawn and on editor save, mapping to curated condition keys (中毒 → `poisoned`). `EffectsCell` flags immune conditions with a non-blocking ⚠ — the condition still applies, DM authority (ADR-0002). Round-trip rule: the editor's R/V/I and 狀態免疫 **text** fields keep the snapshot's own text (qualifiers must survive), and saving re-derives the chip lists one-way (text → chips) — chips never round-trip back to text.

**Traditional Chinese name overlay for the SRD seed (2026-07-08, `1f0fc19`).** The 334 SRD monsters originally seeded English-only (`nameZh: ""`), so they had no Chinese name until spawned and renamed by hand. `gen-enemies.mjs` now carries an `SRD_NAME_ZH` map (each monster's exact English `name` → Traditional Chinese) applied in `mapSrd`; regenerating `enemySeed.ts` names all 334 permanently (guard: 0 blank `nameZh` after regen means every key matched — a misspelled key would leave a blank). These are community-consensus names — there is **no** official WotC Traditional Chinese 5e edition — matching the official Simplified names where they exist; a handful of exotic transliterations (Aboleth 阿伯雷斯, Behir 貝西爾, Glabrezu 格拉布雷祖, …) are the accepted forms and are edited in that one map. Because DBs seeded before this shipped already exist and `seedAll` never touches existing rows, a separate DM-gated `enemies.backfillZhNames` mutation fills `nameZh` from `ENEMY_SEED` (by `seedKey`) **only where the row's `nameZh` is still blank** — hand-edited names are never overwritten (ADR-0002) and re-running is a no-op. The UI exposes it as a self-hiding `補中文名(N)` button in `EnemyDbPanel` (Backstage only) that appears while any template lacks a zh name and vanishes once all are filled. To correct or add SRD zh names later: edit `SRD_NAME_ZH`, regenerate, and (for a live DB) blank the affected row's name so backfill re-fills it — the overlay only ever fills blanks.

Consequence for future work: adding a new template source = extend `gen-enemies.mjs` with a mapper to the same `SeedEnemy` shape (keep its native action JSON as-is) and, if its action shape differs, teach `recipeFromAction` the new shape in that one function. Adding a new damage type or condition immunity means extending the alias map / `parseConditionImmunities` keys in `convex/rules.ts` and `convex/enemyFields.ts` respectively — the schema (`v.array(v.any())` action blocks, free-text R/V/I) does not change.
