# PLAYGROUND_MODE flag and the character portability model

ADR-0014 creates a public playground deployment sharing code with private tables. This ADR fixes *how* one codebase serves both without a fork, and what the character-card data contract becomes. Implementation is specced in `openspec/changes/prep-public-release/`; decision record in `docs/grilling/2026-07-12-grilling-character-creator.md`.

## Decision

1. **One env flag, safe by absence.** `PLAYGROUND_MODE` is a Convex environment variable read **server-side**. Unset/false (the default every self-hoster gets with zero configuration) = exactly today's behavior: characters are global, persistent, editable. Only our demo deployment sets `true`. The frontend learns the mode via a field carried on an existing game query — display only; every enforcement (stamping, read-only, size limits) happens in mutations.
2. **Playground card isolation by `gameId` stamp.** With the flag on, cards created or imported inside a game are stamped with that game's id; `characters.list` returns unstamped (global) cards + the current game's stamped cards. One optional schema field; unstamped = today's cards, zero migration. No promotion mechanism — self-host/table deployments simply never stamp.
3. **Seed cards are read-only on the playground.** Read-onlyness is derived, not stored: `PLAYGROUND_MODE && seedKey !== undefined` (no `locked` field). Sample cards are four SRD 5.1 pregens (fighter/wizard/cleric/rogue) in `convex/demoSeed.ts` — the only character data the public repo ships. The rejection message points at export: "export it as your own card to edit."

   *Amended during implementation.* The sample cards were originally meant to run through the existing CSV → `gen:characters` → seed pipeline. They are hand-written data instead. A generator needs a source, and a source that can ship has to be open content we are licensed to pass on — at which point four cards are less code than the generator would be. `gen:characters` is retired; character data in this repo means SRD open content (see *Consequence for future work*).

4. **Files are the portability contract.** Export/import of cards as JSON with envelope `{format:"war-table-5e-character", version:1, exportedAt, cards:[…]}` — cards always an array (importer takes 1..N), UI operates per-card, filename `<name>.dndcard.json`. Import customs run server-side: envelope check with a human error, field whitelist (strip `seedKey` and unknown keys), size limits, `hp` clamped to `[0,maxHp]`, validator shape. On the demo, export/import IS the persistence story (data wipes periodically); on private deployments it is backup + cross-instance portability.

   *Amended during implementation.* **A card in the file is the whole card, not the `characters` row.** As first written this said `cards:[…]` without saying what a card contains, and the implementation read it as "the card table's columns" — producing files that dropped every recipe, resource pool, and character-owned effect. For the friend group's six cards that was 32 recipes and 9 pools silently absent from their only backup. So the payload is: every `characters` column (minus `seedKey`/`gameId`/`_id`/`_creationTime`) **plus** the character-owned `resources`, `recipes`, and `effects`. Rules that fell out of it:
   - **Recipes name their pool by `label` (`resourceKey`), never by id** — ids mean nothing in another deployment; a label is stable, readable in the file, and already the seed pipeline's idiom. Import creates pools first, then binds. An unresolvable key keeps the recipe and drops the link: an unlinked recipe still rolls, whereas silently binding to the wrong pool corrupts the math.
   - **`null` means absent, not a value.** These tables declare no nullable field (`v.optional` = key missing), but the views they export from render "no value" as `null`, and JSON cannot carry `undefined`. Without normalizing at the boundary, any card with a custom effect round-trips into `Expected string, got null`.
   - **Child fields are enumerated, not passed through.** Convex rejects a document carrying an undeclared field, so an unfiltered stray key fails the whole import rather than being dropped the way a stray card field is.
   - **`tempHp` joins the card view.** Combat's write-through had been writing it to the card while `CharacterView` lacked the field — writable, unreadable, unexportable. It is the exact failure this contract now tests against.
5. **Content size customs everywhere.** One shared validation on create/update/import: ≤30,000 chars per text field, ≤300KB per serialized card. Oversize errors point to the front door: put large reference tables in an external doc and link them from the card.

## Why

- **Forgetting must be safe.** The flag's polarity puts the burden of remembering on us (one person, one deployment) instead of on every future self-hoster; a misconfigured deployment degrades to a normal private table, never to an exposed playground.
- **The global characters table is the demo's only cross-visitor surface.** Games already isolate via tokens; cards were the one thing every visitor shared. Stamping closes that with one optional field, while leaving our own table's cross-game campaign continuity (the schema's own "campaign-state source of truth" contract) untouched.
- **Server-side enforcement because the client is the visitor's.** A visitor can fake any UI locally; only mutations they can't rewrite are a boundary.
- **The size limits are empirically calibrated.** Real cards max out at ~1,264 chars/field; the incident that motivated the limit (a player pasting a ten-thousand-line wild-magic table onto a card, crashing the table's browsers) is ~300k chars. 30k is ~24× real use and ~10% of the pathological case; the Google-Docs-link workaround the table already invented organically is promoted to the official guidance.
- **Envelope versioning is mail to our future selves.** `version: 1` lets the importer accept old files after schema evolution, the same optional-field compatibility discipline the schema already uses.

## Considered options

- **`SELF_HOST=true` flag (inverted polarity).** Rejected: default falls on the dangerous side for strangers.
- **A stored `locked` field on cards.** Rejected (simplification S1): derivable from flag+seedKey; one less field and the table deployment's own seeded six cards stay editable without special-casing.
- **DM "promote to campaign card" button (unstamp gameId).** Rejected (user simplification, grilling Q9): self-host deployments never stamp, so there is nothing to promote; the button only served a hosted-multi-tenant model we rejected in ADR-0014.
- **Standalone homepage character editor (no game context).** Rejected: requires an untokened write endpoint (spam target) and transplanting the editor out of the GameShell/`.wt` theming scope; creating a game first is one click.
- **Per-card single-object file format.** Rejected: the array envelope costs nothing at length 1 and makes party export a free future feature with one importer.

## Consequence for future work

Adding a character field = add it to the validator/whitelist + bump consideration for `version` (importer must keep accepting version 1). **This is enforced, not remembered:** `convex-tests/cardRoundTrip.test.ts` introspects the schema and asserts that every column of `characters`/`recipes`/`resources`/`effects` appears in the file whitelist, so adding a column without extending the format fails the suite immediately. Its companion drives a fully-populated card through the real export path (`list` → `toPortableCard` → JSON → `importCards`) and demands byte-identical return — Markdown included, since the card previews `refs` bodies and `classRules` through SafeMarkdown and an identical string is the only thing that guarantees an identical preview. Both were mutation-tested (add a column; drop a field from the export) and fail as intended. Never read `PLAYGROUND_MODE` in the frontend for enforcement decisions. New shared surfaces (e.g. a future global template library) must answer the same question cards did: what does a stranger see, and who can vandalize it. Phase 2 items (wipe mutation + cron, party export, long-content collapsed rendering) build on these contracts without changing them.

*Amended during implementation.* This ADR originally aimed only to keep the group's own seed out of the demo's seed path. Publishing the repo raised the bar: a deployment is not the unit of exposure — the repository is. Anything a private table writes (its players' cards, their writing) belongs in that table's database, not in the source tree, whichever deployment happens to read it.

So the rule is: **character data in this repo means SRD open content.** The group's own cards live only in their own deployment, and their backup is an exported `.dndcard.json` restorable via Import — which covers every card rather than only the ones a seed file knew about. Tests assert every seeded card's `seedKey` matches `^demo_` (an allowlist: a denylist would have to name the things it excludes, which defeats the point).
